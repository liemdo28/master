## Core Rules

1. **No Artifact = No Validation**
   - Validation requires real artifacts.
   - If artifact missing, status must be FAIL or UNKNOWN.

2. **No Journal = Task Incomplete**
   - Task completion requires journal entries in `master-journal/events`.
   - If journal not found, export must include `UNKNOWN_STATUS.md`.

3. **No Real Data = UNKNOWN**
   - If no real source connected, exporter must mark section as UNKNOWN.
   - Never claim PASS/ACTIVE/APPROVED without evidence.

4. **No Evidence = Cannot Mark PASS**
   - PASS requires verifiable artifacts, logs, manifests, or process checks.

5. **Exporter cannot generate success without source**
   - Hardcoded success text is forbidden.
   - Success must come from real data.

6. **Lifecycle must run validation before export**
   - Required flow: `Lifecycle -> Validation -> Artifact -> Journal -> Export`

7. **Port must be registered before service start**
   - Services must declare/listen ports in registry.
   - Unknown port ownership is governance violation.

8. **Node process must have owner**
   - Each Node.js process should map to a project/service.
   - Unknown owner = audit risk.

9. **Unknown is acceptable; fake PASS is not**
   - UNKNOWN is safer than false success.

10. **CEO approval required for cloud operator actions**
   - Cloud operator / deploy / destructive remote actions require explicit CEO approval.

## Exporter Rules

- `OPEN_BUGS.md`: If no bug tracker source, mark `UNKNOWN - bug tracker not connected.`
- `QA_STATUS.md`: If no QA artifact, mark `UNKNOWN - no latest QA run artifact found.`
- `APPROVAL_STATUS.md`: If no approval registry, mark `UNKNOWN - approval registry not connected.`
- `ERROR_SUMMARY.md`: If no error logs, mark `UNKNOWN - error logs not accessible.`
- `WORKER_STATUS.md`: If no worker monitor, mark `UNKNOWN - worker process not monitored.`
- `ARCHITECTURE.md`: Do not mark engine ACTIVE unless verified.

  - Do not hardcode success
  - Still export package with `UNKNOWN_STATUS.md`
