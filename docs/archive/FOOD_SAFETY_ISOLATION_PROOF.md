# Food Safety Isolation Proof — Phase 21.7

## Date: 2026-06-22

---

## Problem

Food Safety was responding in CEO direct chat with:
```
Mi is not available on this bot.
This bot is only for Food Safety and team support.
```

## Root Cause

`handleImageMessage()` in `message-listener.js` had NO CEO sender check.
When CEO sent any image, food safety pipeline processed it and responded.

## Fix Applied

### `message-listener.js` → `handleImageMessage()`

Added at the very top of the function:

```js
// Phase 21.7: CEO sender isolation — food safety MUST NOT respond in CEO direct chat
const senderPhone = (msg.author || msg.from || '').replace('@c.us', '').replace('@g.us', '');
if (miAccess.isCeoSender(senderPhone) && !chatId.includes('@g.us')) {
  log.info('[MESSAGE_FLOW] food_safety_blocked_ceo_direct_chat', { chatId, senderPhone, messageId });
  return; // CEO direct chat images are NOT food safety scope
}
```

## Isolation Rules

| Condition | Behavior |
|-----------|----------|
| CEO sends image in DM | BLOCKED — silent return |
| CEO sends image in food safety group | ALLOWED — group context is food safety scope |
| Non-CEO sends image in food safety group | ALLOWED — normal food safety flow |
| Any user sends image in non-food-safety group | BLOCKED by FOOD_SAFETY_ENABLED check |

## No "Mi is not available" Text in CEO Chat

The text "Mi is not available on this bot" is NOT in the gateway codebase.
It was likely generated by the food-safety gateway's ConversationRouter when
the CEO sent a non-image text message that fell through to food safety.

With the Phase 21.7 owner lock, CEO messages now claim `mi_core` owner first,
so food safety handlers never see CEO text messages.
