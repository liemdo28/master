## Symptom

After a successful workflow response is delivered to CEO on WhatsApp, a second bubble appears:

```
Mi-Core is temporarily unavailable. Please try again later.
```

This creates a false failure perception despite the workflow completing successfully.

---

## Root Cause Analysis

### Primary Root Cause: External Linked WhatsApp Device

**Confirmed via isolation test (2026-06-16):**

1. Local PM2 `whatsapp-ai-gateway` was stopped completely.
2. A test message was sent from WhatsApp Web.
3. The "Mi-Core is temporarily unavailable" reply **still appeared**.
4. Therefore: an **external linked device, old bot runtime, or remote gateway** attached to the same LD Agent WhatsApp account is generating the fallback message.

Evidence: `E:\Project\Master\mi-core\.local-agent-global\whatsapp-proof\TEST-DEV5-009-local-gateway-stopped-external-failure.png`

### Secondary Root Cause: Self-Chat Reprocessing (FIXED)

The gateway's `message_create` listener allowed `fromMe` messages in self-chat mode. When `replyService.isGatewaySentMessage(msg)` missed identifying a gateway-sent message, the successful workflow reply was re-forwarded to Mi-Core as a new input. The second forward failed (Mi-Core returned error for the garbled re-input), triggering a fallback reply.

**Fixed in:** `E:\Project\Master\whatsapp-ai-gateway\src\whatsapp\message-listener.js`
- Added `isMiExecutionReplyText()` guard (line 248)
- Self-chat messages matching execution signatures are now ignored

### Tertiary Root Cause: No-Prefix Fallback Path (FIXED)

The no-prefix Mi forward path previously sent `forwardResult.reply` even when `forwardResult.ok === false`. Failed forwards could produce a user-facing fallback bubble after a valid first response.

**Fixed in:** `sendMiForwardResult()` function (message-listener.js line 134)
- Failure replies from Mi-Core are logged but **never sent to the user**
- Only `forwardResult.ok === true && forwardResult.reply` triggers outbound send

## Current Protection Stack (Local Gateway)

1. **Response Lock** (`responseLocks` Map, TTL 2 min) — prevents duplicate success sends for same inbound message
2. **Recent Success Cache** (`recentMiSuccesses` Map) — prevents fallback after recent success on same chat+text
3. **Execution Reply Ignore** (`isMiExecutionReplyText()`) — ignores self-echoed execution replies
4. **Failure Reply Suppression** (`sendMiForwardResult()`) — Mi failure replies are logged, never sent
5. **Stale Success Guard** (`latestInboundByChat`) — suppresses late responses for older messages
6. **Outbound Send Guard** (`installOutboundSendGuard()`) — blocks banned text patterns at `client.sendMessage` level
